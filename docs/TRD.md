# TrailGuard AI вҖ” Technical Requirements Document

## 1. Architecture Overview

TrailGuard AI follows a **three-tier monorepo architecture**:

- **Frontend**: Next.js 14 (React) with TypeScript, Tailwind CSS, D3.js
- **Backend API**: FastAPI (Python 3.12) with SQLAlchemy + Alembic
- **Database**: PostgreSQL 16
- **Graph Engine**: NetworkX (in-process graph analysis)
- **Orchestration**: Docker Compose for local development and deployment

## 2. Stack Decisions

| Component | Choice | Rationale |
|-----------|--------|-----------|
| Frontend Framework | Next.js 14 (App Router) | SSR for dashboard performance, file-based routing, React ecosystem |
| Styling | Tailwind CSS | Rapid prototyping, consistent design system, small bundle |
| Graph Visualization | D3.js + Force-Directed Layout | Custom interactive graph rendering, no heavy library dependency |
| Backend Framework | FastAPI | Async support, auto-generated OpenAPI docs, Pydantic validation |
| ORM | SQLAlchemy 2.0 | Mature, async support, Alembic migrations |
| Database | PostgreSQL 16 | JSONB for flexible evidence storage, robust transaction support |
| Graph Analysis | NetworkX | Python-native graph algorithms, sufficient for deterministic detection |
| Auth | JWT + bcrypt | Stateless auth, no session store needed |
| Containerization | Docker Compose | Consistent dev/prod parity, minimal infrastructure requirements |

## 4. API Design Principles

- **RESTful URLs**: `/api/v1/accounts`, `/api/v1/alerts`, `/api/v1/cases`
- **Consistent Responses**: `{ "data": ..., "meta": { "page": ..., "total": ... } }`
- **Error Format**: `{ "detail": "message", "code": "ERROR_CODE" }`
- **Pagination**: Cursor-based for lists, page-based for small collections
- **Authentication**: JWT Bearer token via `Authorization` header
- **Versioning**: URL-prefixed (`/api/v1/`)

## 5. Database Design

### Core Tables

- **accounts**: Account profiles with risk scores and metadata
- **transactions**: Individual financial transactions with source/destination
- **detection_events**: Alerts generated by detection engine
- **analysis_runs**: Execution records of detection pipeline
- **investigations**: Case management records
- **risk_assessments**: Account-level risk evaluations
- **audit_events**: Security and activity audit log
- **users**: System users with role assignments
- **datasets**: Imported dataset metadata

### Key Relationships
- Account вҶ’ Transactions (1:N)
- Detection Event вҶ’ Accounts (M:N)
- Investigation вҶ’ Detection Events (M:N via evidence join)
- User вҶ’ Investigation (1:N)

## 6. Security Requirements

- **Authentication**: JWT with 30-minute access token expiry
- **Password Storage**: bcrypt hashing
- **Authorization**: Role-based access (Analyst, Investigator, Admin)
- **Input Validation**: Pydantic schemas on all API endpoints
- **CORS**: Restricted to configured origins
- **SQL Injection**: Prevented by SQLAlchemy parameterized queries
- **Audit Trail**: All sensitive operations logged in audit_events table
- **Secrets**: All secrets managed via environment variables, never in code

## 7. Deployment Architecture

### Local Development
- Docker Compose with three services: `db`, `api`, `web`
- Hot-reload on both API (uvicorn --reload) and frontend (Next.js dev)
- PostgreSQL with persistent volume

### Production (Target)
- **Database**: Managed PostgreSQL (Railway, Fly.io, or AWS RDS)
- **API**: Containerized FastAPI on Railway/Fly.io
- **Frontend**: Static export or server on Vercel
- **File Storage**: Local or S3-compatible for report files
